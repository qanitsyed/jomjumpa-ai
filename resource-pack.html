// the craft · how to prompt

Prompt like you mean it.

The Golden Rule is one line for one kind of app. Past that, prompting is a craft — and most of the craft is knowing what not to ask for in a single shot. Here's the short version.

The single biggest mistake
Don't one-shot a complex app. Ask for the smallest thing that works, then add one feature at a time.

When you describe a five-feature app in one prompt, the model hallucinates features, regenerates everything on each iteration (losing your edits), and writes 500 lines you can't debug. The smaller the first prompt, the faster everything after it goes. If you can't describe the app in one paragraph, your prompt is too big.

By the type of app you're building.

Five shapes cover almost everything you'll try tonight — and almost everything the room said they wished they had. The pattern changes with the shape.

Calculators & lookups small · one-shot OK
Zakat calc. Tax estimator. Shariah-fund check.
// expense calc · shariah compliance check · CV reformatter · stock screener

One prompt is fine. Specify the inputs, the outputs, and one worked example. End with the Golden Rule. These apps live or die on edge cases — list the weird ones in the prompt before you write a line of code.

Trackers & dashboards medium · two prompts
Monthly report. Net worth tracker. Team performance.
// productivity tracker · billable-hours autolog · project dashboard · expense tracker

Data first, then the UI. Prompt one: "Here's the shape of one row, here's 5 dummy rows — give me the data structure and localStorage code." Prompt two: "Now build the dashboard on top of that data." Skipping the first prompt is why your dashboard keeps re-rendering wrong.

Intake & submission tools medium · stage the storage
Sign-up forms. Feedback collectors. Application intake.
// event RSVP · client onboarding form · survey · waitlist

Don't try to wire a real database in one prompt. Start with localStorage (works offline, shows the flow). Upgrade to a Google Sheet via a webhook for v2. Reach for Supabase only when you actually need multi-user reads — that's Track 4 territory, not the first hour.

Generators & summarizers medium · use a builder
Blog generator. Market-insights digest. LinkedIn drafter.
// auto blog publisher · contract-clause flagger · meeting summarizer · CV rewriter

These need an LLM API call from inside your code — and a single HTML file can't safely hold the API key. Use Lovable or Bolt; they handle the key and the env for you. Prompt: "Build me a [thing]. When the user clicks Generate, call Claude with this system prompt: ..."

Email / calendar / CRM agents large · out of scope
Inbox manager. Calendar agent. Day-to-day startup AI.
// agentic admin · auto-reply triage · CRM updater · meeting prep agent

Don't try to build this in one hour. Real agents need auth, real APIs, and real money on a real timer. Build the dashboard that shows what you'd automate, then wire the automation in Make.com or n8n once the dashboard works. Half the value is just seeing the workflow on a screen.

When in doubt smaller · always
Pick the smallest shape that proves the idea.
// ship one feature · iterate · then add the next

If your idea spans two of the shapes above, pick one and ship it tonight. A working tracker beats a half-built agent. Half the things the room asked for are agents — but the calculator inside the agent is what you can actually finish before 9pm.

Six prompt hacks that just work.

Universal patterns. Use them across any tool — Claude, ChatGPT, Lovable, Bolt, Cursor. They don't replace thinking; they prevent the most common ways prompting wastes your time.

Anchor with one worked example.
Paste one sample input and the exact output you want. The model copies the shape. Five sentences of description loses to one concrete example every time.
Specify the stack — every time.
End every build prompt with the Golden Rule: single self-contained HTML file, no external libraries, clean modern design, mobile-friendly. Without it the model defaults to React + a build step you don't have.
Constrain what not to do.
"Don't ask me clarifying questions — make sensible defaults." "Don't add a backend." "Don't use frameworks." Negative constraints save more time than positive ones because they cut off the dead branches first.
Ask for tradeoffs before the code.
"Before writing code, give me 3 ways to structure this and the tradeoff for each." You'll catch the wrong direction in 30 seconds instead of in 500 lines. Especially for anything involving storage, auth, or layout.
Paste errors verbatim.
Copy the full error message — stack trace, line number, the lot. Don't paraphrase, don't summarize. The model debugs from the literal string. "It's broken" gets you guesses; the raw error gets you the fix.
Re-prompt. Don't keep chatting.
Every 6–8 messages, start a fresh chat. Paste your current working code, then your next ask. Long threads drift — the model starts contradicting earlier decisions and "fixing" things you didn't ask it to touch.
